Plain-English: why the table is confusing — and what makes it make sense

Why it's confusing right now

  1. The “Group” button doesn't sort — it clusters. It just stacks each child under its parent; afterward the page even tells a screen reader the column is “not sorted,” so nothing seems to happen.
  2. Sorting any normal column scatters the children. A child only sits next to its parent while “Group” is on. Sort by Status, Title, anything else, and children drift away from their parent.
  3. A filter can leave a child with no parent on screen — silently. If a filter matches a child but not its parent, the parent vanishes and nothing says “the parent was filtered out.” You can't tell whose child it is.
  4. “P” and “C” tell you nothing. No project id, no parent name — and because every project has its own unique number (children aren't numbered off the parent), the id can't tell you who's related either.

Under the hood: no “row name” so a screen reader never says which project a cell belongs to; every cell wrongly points at the empty filter box; standalone projects show a blank cell.

What makes it make sense

  1. Give every row its project id as its “name” (a row header) — then any cell tells you which project it belongs to.
  2. Always name the parent in plain words, and flag when it's gone — “child of [parent],” and “parent filtered out” when it's not shown. Two clean ways below: an inline note (Table 2) or a dedicated Relationship column (Table 4) — the column tested clearest.
  3. Say what happened — announce “Showing 7 of 60” on filter, and show “No projects match” when nothing does.
  4. Call “Group” what it is (grouping, not sorting) and report the honest state.
  5. Mark standalone projects (“standalone — not part of a group”) instead of a blank cell.

Bottom line: it's not confusing because it's complicated — it's confusing because the relationships and states aren't spelled out. Spell them out and it makes sense. The five tables below show each piece.

Table 1 — parent kept as a “context” row (current behaviour)

A filter match on a child but not its parent keeps the parent visible as a dimmed context row. Try: Status → Released.

Table 2 — Option A: parent dropped, child shown as an inline orphan

What changes: when a filter matches a child but not its parent, the parent is not shown. The child appears on its own and carries the whole story:

  1. Names the missing parent — “child of WDAS-1001”, so its lineage survives.
  2. Says the filter caused it — “parent filtered out” (parity with what a sighted user sees).
  3. Announces it on filter — “Showing 4 of 11 — 2 are child projects whose parent is filtered out.”

Try: Status → Released. Compare with Table 1.

Table 3 — Option C: filters in a left drawer

What changes: the in-table filter row is gone — filters move to a left-hand drawer with a search field, checkbox groups (multi-select), a radio group (single-select), and a text input (the layout your team is weighing). The table itself is unchanged (accessible expand/collapse + inline orphans). Each control group is a labelled <fieldset>; the drawer toggle uses aria-expanded / aria-controls.

Try: check Status “Released” in the drawer to see the inline orphans, or type in Search.

Table 4 — Option B: dedicated Relationship column

What changes: instead of an inline marker, every child row has a dedicated Relationship column (first column) that always names its parent — even when the parent is filtered out (shown as “WDAS-1001 — filtered out”). Because it’s a real column, a screen reader reads it as “Relationship: WDAS-1001 — filtered out” via the column header, and the association survives any filter or sort. Tradeoff: it adds a column to an already-wide table (the concern your team raised).

Try: Status → Released. The parent row is gone; each orphan names its parent in the Parent column.

Table 5 — the current state: Group shows only “P” / “C”

This is what’s shipping today. The Group column shows just “P” for parent and “C” for child — no project id, no “child of …”. So you can’t tell which project a row is, or which parent a “C” belongs to — and a screen-reader user simply hears “P” or “C”.

Try: filter Status → Released. The parent (“P”) is filtered out and gone, leaving bare “C” rows with no parent anywhere in view — you can’t tell whose children they are. Compare with Tables 1–4, where the id + “child X of Y, under [parent]” keep it clear.
