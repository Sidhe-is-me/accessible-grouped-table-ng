Real Angular build. Multi-select filters use @angular/cdk/listbox; sorting/counts announce via Angular CDK LiveAnnouncer. Two tables below show two ways to handle an orphaned child — a child that matches a filter when its parent does not.

Table 1 — parent kept as a “context” row (current behaviour)

When a filter matches a child but not its parent, the parent stays visible as a dimmed context row, so the child keeps its place under it. Try: open Status → Released — “Aurora Falls (franchise)” stays as a context row above its two released children.

Table 2 — Option A: parent dropped, child shown as an inline orphan

What changes: when a filter matches a child but not its parent, the parent is not shown at all. The child appears on its own and carries the whole story so nothing is lost:

  1. Names the missing parent — the row reads (and the screen-reader name says) “child of WDAS-1001”, so its lineage survives even though the parent row is gone.
  2. Says the filter caused it — “parent filtered out” (not just “not shown”), so a screen-reader user gets the same information a sighted user gets from seeing no parent row.
  3. Announces it on filter — the live region says e.g. “Showing 4 of 11 — 2 are child projects whose parent is filtered out.”

Try: open Status → Released. The parent “Aurora Falls (franchise)” (In Production) is gone; “Aurora Falls” and “Aurora Falls II” appear as inline orphans that still name their parent. Compare with Table 1.
